# `@ggpwnkthx/csr`

Client build and development workflow tooling that wraps `@ggpwnkthx/esbuild`.

## Overview

This package provides:

- **Production client bundle generation** with content-hashed assets
- **Development watch mode** with live reload scaffolding
- **Deterministic manifest generation** for server/runtime consumption

## Required Permissions

| Operation        | Required Flags                                                |
| ---------------- | ------------------------------------------------------------- |
| Production build | `--allow-read`, `--allow-write`, `--allow-run`                |
| Development mode | `--allow-read`, `--allow-write`, `--allow-net`, `--allow-run` |

## Installation

```typescript
import { buildClient, devClient, readManifest } from "jsr:@ggpwnkthx/csr@^0.1";
```

## Usage

### Production Build

```typescript
import { buildClient } from "jsr:@ggpwnkthx/csr@^0.1";

const result = await buildClient({
  entryPoints: ["src/client.ts"],
  outdir: "dist",
  sourcemap: false,
});

console.log("Output files:", result.outputFiles);
console.log("Manifest:", result.manifestPath);
```

### Development Mode

```typescript
import { devClient } from "jsr:@ggpwnkthx/csr@^0.1";

const dev = await devClient({
  entryPoints: ["src/client.ts"],
  outdir: ".dev",
  port: 35729,
});

console.log(`Dev server running on http://${dev.hostname}:${dev.port}`);

// Clean shutdown
await dev.stop();
```

**Dev server behavior:**

- Serves all files from `outdir` as static files
- Provides virtual URLs for accessing built assets:
  - `GET /@entry/<name>` - redirects to the current JS for that entry
  - `GET /@style/<name>` - redirects to the current CSS for that entry (if any)
- Injects live reload script into HTML responses (`<script>new EventSource("/~livereload").onmessage=()=>location.reload()</script>`)
- Injects before `</body>` if present, otherwise appends to document
- Falls back to `index.html` for directory requests (e.g., `/foo/` → `/foo/index.html`)
- Watches parent directories of entry points for changes
- Rebuilds on file changes and sends `data: reload\n\n` via SSE to `/~livereload`

**Note:** HTML files must exist in `outdir` before starting the dev server (either generated by the build or copied there beforehand).

### Reading a Manifest

```typescript
import { readManifest } from "jsr:@ggpwnkthx/csr@^0.1";

const manifest = await readManifest("dist/manifest.json");

for (const [entryName, entry] of Object.entries(manifest.entries)) {
  console.log(`${entryName} -> ${entry.outputFile} (${entry.hash})`);
}
```

**Manifest path:** The manifest is always written to `{outdir}/manifest.json`.

## API

### `buildClient(options)`

Performs a production client build.

Production builds use the following managed settings (cannot be overridden via `esbuildOptions`):

- `platform: "browser"` - Browser-oriented output
- `format: "esm"` - ES module output
- `splitting: true` - Code splitting enabled; shared code is extracted into separate chunks
- `entryNames: "[dir]/[name].[hash]"` - Hashed filenames for entry bundles
- `assetNames: "[dir]/[name].[hash]"` - Hashed filenames for assets
- `chunkNames: "[dir]/[name].[hash]"` - Hashed filenames for shared chunks
- `minify: true` - Minification enabled
- `bundle: true` - Bundling enabled
- `write: true` - Files written to outdir
- `metafile: true` - Metadata collected for manifest generation

**Options:**

```typescript
interface BuildClientOptions {
  /** One or more entry files (required) */
  entryPoints: string | string[];
  /** Output directory (default: "dist") */
  outdir?: string;
  /** Root directory for resolving entry points (default: Deno.cwd()) */
  rootDir?: string;
  /**
   * Additional esbuild options merged with defaults.
   * Cannot override: entryPoints, outdir, bundle, write, format, entryNames,
   * assetNames, chunkNames, platform, splitting, minify, metafile, sourcemap,
   * define, plugins
   */
  esbuildOptions?: Record<string, unknown>;
  /** Generate manifest file (default: true) */
  manifest?: boolean;
  /** Emit source maps (default: false) */
  sourcemap?: boolean;
}
```

**Returns:**

```typescript
interface BuildResult {
  /** Relative paths to output files within outdir */
  outputFiles: string[];
  /** Absolute path to manifest.json, or null if manifest: false */
  manifestPath: string | null;
  /** Build warnings with message and optional location */
  warnings: BuildWarning[];
}
```

### `devClient(options)`

Starts a development server with watch mode and live reload.

**Options:**

```typescript
interface DevClientOptions {
  /** One or more entry files (required) */
  entryPoints: string | string[];
  /** Output directory (default: ".dev") */
  outdir?: string;
  /** Port for dev server (required) */
  port: number;
  /** Additional esbuild options merged with defaults */
  esbuildOptions?: Record<string, unknown>;
}
```

**Returns:**

```typescript
interface DevHandle {
  /** Server hostname (always "localhost") */
  hostname: string;
  /** Server port */
  port: number;
  /** Resolved absolute path to outdir */
  outdir: string;
  /** Stops the dev server and cleans up watchers */
  stop: () => Promise<void>;
}
```

### Manifest Schema

```typescript
interface AssetManifest {
  version: 1;
  /** ISO 8601 timestamp */
  timestamp: string;
  /** Keyed entries by source path (entry points, imports) */
  entries: Record<string, ManifestEntry>;
  /** Unkeyed outputs (shared chunks, generated assets) */
  assets?: UnkeyedOutputEntry[];
}

interface ManifestEntry {
  /** Original source path */
  originalPath: string;
  /** Relative path from outdir to output file */
  outputFile: string;
  /** File size in bytes */
  size: number;
  /** SHA-256 hex digest of file contents (64 characters) */
  hash: string;
  type: "js" | "css" | "asset";
}

interface UnkeyedOutputEntry {
  /** Relative path from outdir to output file */
  outputFile: string;
  /** File size in bytes */
  size: number;
  /** SHA-256 hex digest of file contents (64 characters) */
  hash: string;
  type: "js" | "css" | "asset";
  kind: "chunk" | "asset";
}
```

**Keying strategy:**

- Entry point outputs are keyed by their `entryPoint` path
- CSS imports and single-source assets are keyed by their source path (best-effort)
- Shared chunks and truly unkeyable outputs go to `assets[]` with `kind: "chunk"` or `kind: "asset"`

**Path normalization rules:**

- `outputFile` is always relative to `outdir` (never absolute)
- `outputFile` uses forward slashes only (no backslashes)
- `entries` keys are sorted alphabetically

### Manifest Utilities

```typescript
// Read and validate a manifest
const manifest = await readManifest("dist/manifest.json");

// Resolve the expected manifest path for an outdir
const manifestPath = resolveManifestPath("dist"); // "dist/manifest.json"

// Write a manifest to an outdir
const writtenPath = await writeManifest(manifest, "dist");
```

### Error Handling

```typescript
import { buildClient, BuildError } from "jsr:@ggpwnkthx/csr@^0.1";

try {
  await buildClient({ entryPoints: "src/client.ts" });
} catch (err) {
  if (err instanceof BuildError) {
    console.error("Build failed:", err.message);
  }
}
```

**Error classes:**

| Class                       | Thrown When                            |
| --------------------------- | -------------------------------------- |
| `BuildError`                | esbuild build fails                    |
| `DevServerError`            | Dev server fails to start              |
| `ManifestError`             | Manifest read/write validation fails   |
| `ValidationError`           | Option validation fails                |
| `EntryPointValidationError` | Entry point file does not exist        |
| `OutdirValidationError`     | outdir is empty or invalid             |
| `PortValidationError`       | Port is not an integer between 1-65535 |

## Architecture

This is a monorepo containing multiple packages:

```
packages/
├── shared/           # Shared utilities (errors, hashing, validation)
│   └── src/mod.ts
├── manifest/         # Manifest types, generation, I/O, validation
│   └── src/mod.ts
├── build/            # Production build orchestration
│   └── src/mod.ts
└── dev/              # Development server with live reload
    └── src/mod.ts
```

The main `@ggpwnkthx/csr` package re-exports all sub-packages:

```typescript
export * from "@ggpwnkthx/csr-build";
export * from "@ggpwnkthx/csr-dev";
export * from "@ggpwnkthx/csr-manifest";
export * from "@ggpwnkthx/csr-shared";
```

### Internal Domain Boundaries

| Domain      | Responsibility                           | Public API                                |
| ----------- | ---------------------------------------- | ----------------------------------------- |
| `build/`    | Production build orchestration           | `buildClient()`, `BuildError`             |
| `dev/`      | Development server with live reload      | `devClient()`, `DevServerError`           |
| `manifest/` | Asset manifest schema and I/O            | `readManifest()`, `writeManifest()`, etc. |
| `shared/`   | Internal utilities (hashing, validation) | Not exported publicly                     |

### Subpath Exports

Each sub-package can be imported independently:

```typescript
import { buildClient } from "jsr:@ggpwnkthx/csr-build";
import { devClient } from "jsr:@ggpwnkthx/csr-dev";
import { readManifest } from "jsr:@ggpwnkthx/csr-manifest";
import { hashFile, validateBuildOptions } from "jsr:@ggpwnkthx/csr-shared";
```

## Package Comparison

This repository contains multiple packages:

### `@ggpwnkthx/csr`

**Use this for:** Application build/dev workflows.

- Production client bundle generation with `buildClient()`
- Development watch mode with `devClient()`
- Manifest I/O utilities

```typescript
import { buildClient, devClient } from "jsr:@ggpwnkthx/csr";
```

### `@ggpwnkthx/csr-manifest`

**Use this for:** SSR, deployment tooling, runtime asset lookups, test harnesses.

- Self-contained manifest solution
- No dependency on `@ggpwnkthx/csr-build` or build orchestration
- Can be used independently of the CSR tooling

```typescript
import { buildManifest, readManifest } from "jsr:@ggpwnkthx/csr-manifest";
```

**When to use `@ggpwnkthx/csr-manifest` directly:**

- Building SSR solutions that consume build manifests
- Deployment scripts that need manifest validation
- Test harnesses that validate generated asset maps
- Runtime asset lookup utilities
