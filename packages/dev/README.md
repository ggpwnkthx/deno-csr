# `@ggpwnkthx/csr-dev`

Development server with watch mode and live reload injection.

## Installation

```typescript
import { devClient, DevServerError } from "jsr:@ggpwnkthx/csr-dev@^0.1";
```

## Usage

### Starting a Dev Server

```typescript
import { devClient } from "jsr:@ggpwnkthx/csr-dev";

const dev = await devClient({
  entryPoints: ["src/client.ts"],
  outdir: ".dev",
  port: 35729,
});

console.log(`Dev server running on http://${dev.hostname}:${dev.port}`);
console.log(`Output directory: ${dev.outdir}`);

// Clean shutdown on Ctrl+C
await dev.stop();
```

### Accessing Built Assets via Virtual URLs

The dev server provides virtual URLs for accessing built assets without hardcoded paths:

```
GET /@entry/<name>  → redirects to the JS file for that entry
GET /@style/<name>  → redirects to the CSS file for that entry (if any)
```

For example, if your entry is `src/client.ts`, use `/@entry/client` to get the current JS.

### Live Reload

The server injects live reload script into HTML responses automatically:

```html
<script>
  new EventSource("/~livereload").onmessage = () => location.reload();
</script>
```

The script is injected before `</body>` if present, otherwise appended to the document.

### SSE Stream for Custom Reload Handling

Connect to `/~livereload` to receive reload events:

```typescript
const es = new EventSource("/~livereload");
es.onmessage = (event) => {
  if (event.data === "reload") {
    // Custom reload logic, e.g., update state without full reload
    console.log("Files changed, reloading...");
  }
};
```

## API

### `devClient(options)`

**Options:**

```typescript
interface DevClientOptions {
  entryPoints: string | string[];
  outdir?: string;
  port?: number;
  esbuildOptions?: Record<string, unknown>;
}
```

**Returns:**

```typescript
interface DevHandle {
  hostname: string;
  port: number;
  outdir: string;
  stop: () => Promise<void>;
}
```

## Notes

- HTML files must exist in `outdir` before starting the server (either generated by build or copied beforehand)
- Dev mode uses `splitting: true` and hashed filenames like production for consistency
- Source files are watched recursively in the entry point directories
- The server handles directory requests by falling back to `index.html`

## Permissions

Requires `--allow-read`, `--allow-write`, `--allow-net`, `--allow-run`.
